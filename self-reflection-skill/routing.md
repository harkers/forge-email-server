# Routing — Where Reflections Go

## Rule of Scope

Route the lesson to the narrowest useful scope.

### Use `none`
When the observation is one-off, trivial, or not reusable.

### Use `corrections.md`
When the lesson is general, recurring, or not limited to a single domain or project.

Examples:
- overexplaining answers
- skipping a verification step
- burying the conclusion too late

### Use `domains/<domain>.md`
When the lesson applies to a discipline or task family.

Examples:
- `domains/communication.md`
- `domains/coding.md`
- `domains/privacy.md`
- `domains/design.md`

### Use `projects/<project>.md`
When the lesson only matters inside one named project, product, or client context.

Examples:
- a project-specific phrasing pattern
- a client-specific reporting structure
- an implementation rule unique to one codebase

## Tie-Break Rule

If unsure between global and scoped:
- default to domain rather than global
- default to project rather than domain when the lesson is clearly local

## Promotion Rule

`self-reflection` should not promote directly.
It should send the candidate lesson to `self-improving`, which can track recurrence and promote after repeated successful application.
