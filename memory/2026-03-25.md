## Control-plane orchestration test (CP-002)

Successfully ran deployment diagnosis through control-plane.

**Task:** Diagnose Forge Pipeline port conflict

**Decomposition:**
- Dispatched deployment-diagnosis-agent (spawned)
- Worker failed (no output) → manager took over diagnosis
- Evidence gathered: port listeners, container configs, nginx configs

**Root cause identified:**
- Port 4173 is NOT occupied by host nginx
- Port 4173 is occupied by `openclaw-dynamic-ddh-web` container running in host network mode
- Port 4174 is correctly assigned to `forge-pipeline-web`
- Both services share the same API backend (port 4181)
- Original "port conflict" documentation was stale — issue was already resolved

**Evidence:**
- `ss -tlnp` showed nginx PID 1701743 listening on 4173
- `docker inspect` confirmed host network mode
- Container nginx config showed `listen 4173;`
- API responses showed different project counts (12 vs 14), confirming separate services

**Actions:**
- Updated REVIEW_IMPROVEMENTS.md: removed stale "port conflict" from critical issues
- Updated DEPLOY.md: corrected port documentation, removed misleading instructions
- Added port configuration note explaining DDH/Forge Pipeline architecture

**Commit:** `766cf91` — Fix port documentation: 4173 is DDH, 4174 is Forge Pipeline

**Verdict:**
- Deployment diagnosis model validated
- Worker spawn worked but produced no output (model issue)
- Manager successfully diagnosed independently
- Documentation updated with evidence-first approach

**Lessons:**
- Subagent spawn worked but model (deepseek-r1:14b) produced empty output
- Thinking models (deepseek-r1, phi4-reasoning, gpt-oss, qwen3-thinking) produce special tokens before final answer
- Subagent harness captures raw output but may not extract structured content correctly
- **Fix:** Use standard output models for subagents (gemma3:12b, qwen3:14b, qwen2.5-coder)
- Control-plane correctly rejected empty handoff
