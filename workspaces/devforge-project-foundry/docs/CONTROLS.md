# DevForge: Project Foundry
## Control Model

Status: Planning  
Date: 2026-03-28

## Purpose

Define the top-line controls that govern Project Foundry from strategy through closure.

## Top-Line Controls

| ID | Control | Purpose |
|---|---|---|
| GOV-01 | Governance | Clear ownership, approvals, gates, and decision rights |
| DOC-01 | Documentation and Evidence | Required artifacts are created, linked, and preserved |
| PRIV-01 | Privacy and Data Protection | Privacy by design, minimisation, handling, and review expectations |
| SEC-01 | Security | Security baseline, access control, secrets handling, and secure implementation |
| RSK-01 | Risk Cycle | Risk identification, evaluation, mitigation, and history |
| COM-01 | Communications and Stakeholder Control | Right information to the right people at the right time |
| COM-02 | Distillation and Handoff Communication Control | Outputs are refined and packaged before handoff |
| TRN-01 | Training and Enablement | Approved delivery truth is turned into role-based enablement assets |
| META-01 | Metadata and Classification | Core assets carry structured business and control metadata |
| DLP-01 | DLP and Access Boundaries | Sharing and visibility governed by boundary and sensitivity rules |
| CHG-01 | Build and Change Control | Work changes only through governed paths |
| QA-01 | Quality and Assurance | Output is checked against expectations and acceptance criteria |
| AUD-01 | Audit Trail and Traceability | History of actions, decisions, and handoffs is preserved |
| OWN-01 | Operational Ownership | Named ownership exists before and after release |
| AI-01 | AI and Automation Governance | AI usage, boundaries, and review expectations are controlled |

## Principle

Project Foundry should not rely on good intentions alone. Controls must be explicit, durable, and reflected in documents, roles, and lifecycle gates.
