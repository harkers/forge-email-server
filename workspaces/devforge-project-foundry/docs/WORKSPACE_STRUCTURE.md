- `forge-governance` — approval gates, governance checks, control readiness
- `forge-risk` — risk identification, evaluation, mitigation, history
- `forge-document-engine` — canonical records, templates, evidence, linked documents
- `forge-orchestrate` — scaffolding, repo bootstrapping, structural foundation
- `forge-pipeline` — workflow state, comments, status, traceability
- `forge-whats-next` — planning, dependency sequencing, next-work decisioning
- `forge-control-plane` — execution of approved work packages
- `forge-review` — functional, quality, security, and privacy review
- `forge-release` — release readiness and rollback planning
- `forge-deploy` — deployment execution
- `forge-validate` — post-deploy validation and operational confirmation
- `forge-comms` — communications, distillation, stakeholder handoff packs
- `forge-training` — training and enablement asset generation
- `forge-close` — closure, final evidence, handover
- `forge-fix` — remediation and governed re-entry

## Principle

Project Foundry should separate specialist execution domains clearly enough that each one can later carry its own directives, templates, and AI-assisted role support.
